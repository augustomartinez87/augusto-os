acceptance:
  - Existe un script de seed ejecutable por comando (ej. `npm run seed:spensiv`) que puebla el dev DB de Spensiv en Neon con datos de prueba representativos según el schema materializado (APs, prestatarios, préstamos/cuotas y score snapshots), sin requerir edición manual de archivos
  - El seed es idempotente — correrlo dos veces seguidas no duplica filas (usa upsert por claves naturales); un test/aserción verifica que el conteo de filas es igual tras la 2ª corrida
  - El script aborta con un error claro y NO escribe nada si `DATABASE_URL` no apunta al dev DB de Neon (guard de entorno anti-prod), de modo que nunca pueda correr contra producción
  - Los guard tests existentes de Spensiv (3/3) siguen pasando tras correr el seed
  - Los datos sembrados son sintéticos/ficticios (sin PII real ni datos de clientes reales)
  - Typecheck, lint y tests pasan sin errores

## Contexto

Spensiv corre su DB de desarrollo en **Neon**, con el secreto de conexión en `orchestrator/.env` (gitignored) y el **schema ya materializado el 2026-06-19**, con **3/3 guard tests** pasando (ver memoria `project_spensiv_neon_dev`). Hoy el dev DB está materializado pero sin datos de prueba consistentes: cada quien arma fixtures a mano, lo que hace frágil verificar features que dependen de datos (p. ej. el AP Score badge/gate de F-0001 y F-0002, que leen `ApScoreSnapshot` y modelos de préstamo en Prisma).

Esta feature provee un **seed reproducible e idempotente** para que cualquier loop/dev tenga el mismo set de datos base contra el cual probar, sin tocar prod ni el schema.

El loop debe **REUSAR**, no reinventar:
- El **Prisma client y los modelos ya materializados** (incluido `ApScoreSnapshot` referenciado en F-0001) — el seed se tipa contra esos modelos, no define los suyos.
- El patrón de **guard de entorno** que ya usan los 3 guard tests de Spensiv: reutilizar esa lógica (o el helper que valida que la conexión es la de Neon/dev) para el chequeo anti-prod del seed, en vez de escribir una validación nueva.
- El secreto de conexión vive en `orchestrator/.env` (gitignored) — el script lo lee de ahí, no se hardcodea.
- Convención de columnas **camelCase sin `@map`** del schema existente al construir las fixtures.

## Pasos sugeridos (el planner puede refinar)

1. Crear un módulo de fixtures tipado (ej. `seed/spensiv.fixtures.ts`) con datos sintéticos representativos (varios APs, prestatarios, préstamos/cuotas y score snapshots), tipado contra los modelos Prisma; typechea solo.
2. Crear un helper de guard de entorno (ej. `seed/assertDevDb.ts`) que reuse la lógica de los guard tests para confirmar que `DATABASE_URL` es el dev DB de Neon y lance error si no; typechea solo.
3. Crear el script de seed (ej. `seed/seed-spensiv.ts`) que llame al guard, instancie el Prisma client y haga **upsert** de cada fixture por su clave natural (idempotente).
4. Agregar el comando `seed:spensiv` a `package.json` que ejecute el script.
5. Agregar un test que corra el seed dos veces y asegure que el conteo de filas no cambia entre la 1ª y la 2ª corrida (prueba de idempotencia), y que los 3 guard tests existentes siguen verdes.
6. Documentar el uso del comando y el requisito de tener `orchestrator/.env` con la conexión de Neon en el README/doc correspondiente del orquestador.

## Fuera de alcance

- Cualquier **migración de schema** o cambio de estructura del DB (el schema ya está materializado; el seed solo inserta datos).
- Sembrar o tocar el **DB de producción** de Spensiv.
- Datos **reales o con PII** de clientes; solo datos sintéticos.
- Cualquier **UI** o pantalla (esta feature es `ui: false`).
- Seeds para `kredy` o `argos`; el alcance es exclusivamente Spensiv.
- Ejecutar el seed automáticamente en CI/CD o en cualquier pipeline desatendido.

## Restricciones clave

- **Toca DB → gate humano requerido antes de correr el seed o cualquier migración.** El script no se ejecuta de forma desatendida; un humano lo dispara contra el dev DB.
- El seed **debe abortar sin escribir** si la conexión no es el dev DB de Neon (guard anti-prod obligatorio). Nunca debe poder correr contra producción.
- **Sin migración de schema**: el seed solo inserta/actualiza datos sobre el schema ya materializado (2026-06-19).
- **Idempotencia obligatoria**: usar upsert por clave natural; correrlo N veces deja el DB en el mismo estado.
- Respetar la convención de columnas **camelCase sin `@map`** del schema existente.
- El secreto de conexión permanece en `orchestrator/.env` (gitignored); no se commitea ni se hardcodea.
