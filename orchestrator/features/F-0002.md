## Contexto

La capa AP ya tiene `getLatestApScore(db, agentId)` en `server/services/ap-score.service.ts` (devuelve `number | null`) y la config de score en `DEFAULT_CONFIG` / `loadScoreConfig` (se persiste como JSON en `ApScoreConfig`, columnas `weights`/`targets` — JSON, sin schema rígido).

`preApprove` está en `server/routers/ap.ts` (~línea 340). Ya tiene un gate existente: `checkDebtorLimit` (línea ~426) que, según el `enforcementMode` de `RiskConfig` ('soft' = `requiresManualReview = true`, 'hard' = bloquea), decide. Queremos sumar un gate análogo basado en el score del AP, reusando ese mismo patrón.

## Pasos sugeridos (el planner puede refinar)

1. Agregar un umbral (`minApScore`, default 40) a la config de score (en el JSON de `ScoreConfig` / `DEFAULT_CONFIG` de `ap-score.service.ts`). Sin migración de schema.
2. En `preApprove`, obtener el score del AP con `getLatestApScore`. Si `score != null` y `score < umbral`, aplicar el gate.
3. Reusar el patrón de `checkDebtorLimit` / `enforcementMode`: 'hard' → bloquear/rechazar la pre-aprobación; 'soft' → `requiresManualReview = true`.
4. Si `getLatestApScore` devuelve `null` (AP sin snapshot), no bloquear (no penalizar AP nuevo).
5. Agregar test cubriendo: score bajo + hard → bloquea; score bajo + soft → flag; score null → pasa; score alto → pasa.

## Fuera de alcance

- Migración de schema (el umbral va en el JSON de config existente).
- UI para configurar el umbral.
- Recálculo o histórico del score.

## Restricción clave

**Sin migración de schema.** El umbral vive en el JSON de config. Esto mantiene el feature 100% corrible por el loop contra el sandbox Neon, sin necesidad de gate de producción.
