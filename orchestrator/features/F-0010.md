  - Hay un test (vitest) que verifica, SIN tocar la DB, la lógica pura del seed— (a) el guard de URL rechaza `ep-floral-mud` y acepta la URL de dev, y (b) los builders de datos producen registros con los campos requeridos por el schema (`Transaction.id` no vacío, `closingDay`/`dueDay` en rango, montos parseables como Decimal, `Category` con `name` único por usuario)
  - Typecheck, lint y tests pasan sin errores

## Contexto

El schema de Spensiv ya está materializado en la Neon de dev `spensiv-dev` (`ep-restless-flower-adn0brum`, ver `targets/targets.json`), pero la base está **vacía**— no hay datos para testear el loop ni la app manualmente. Esta feature crea un script de seed que llena esa DB de dev con un dataset coherente y realista (un usuario demo con tarjetas, categorías, gastos e ingresos) para poder validar dashboards, ciclos de facturación y proyecciones a mano.

El loop debe **REUSAR**, no reinventar:
- `prisma/schema.prisma` como fuente de verdad de modelos y campos. Los nombres son camelCase sin `@map` a nivel de propiedad y las tablas usan `@@map` (`users`, `credit_cards`, `transactions`, etc.). Modelos a poblar— `User`, `UserPreferences`, `CreditCard`, `Category`, `SubCategory`, `Transaction`, `Income`. **Ojo con `Transaction`**— su `id` NO tiene `@default`, hay que pasarlo siempre.
- El cliente Prisma ya existe en `lib/prisma.ts` (export `prisma`). El seed puede reusar ese singleton o instanciar su propio `PrismaClient`; en ambos casos la URL sale de `DATABASE_URL`/`DIRECT_URL` (ver `datasource db` en el schema).
- `decimal.js` (`Decimal`) — ya es dependencia — para los montos `@db.Decimal(12,2)`, y `date-fns` — ya es dependencia — para fechas de compra/vencimiento coherentes, sin agregar libs de fechas nuevas.
- El patrón de guard anti-prod del orquestador (`prodDbPatterns: ["ep-floral-mud"]` en `targets/targets.json`)— replicá la misma idea dentro del seed como defensa en profundidad del lado de Spensiv.
- El runner de seeds de Prisma— `prisma db seed` lee `package.json > prisma.seed`. Como el repo no tiene `tsx`/`ts-node`, es admisible sumar `tsx` como devDependency SOLO para ejecutar el `seed.ts` (es imprescindible para correr un seed en TS); no agregar otras dependencias.

## Pasos sugeridos (el planner puede refinar)

1. Crear `prisma/seed-guard.ts` con una función pura `assertNotProdDatabaseUrl(url?: string): void` que lance un error claro si `url` es falsy o contiene `ep-floral-mud`. Typechea solo.
2. Crear `prisma/seed-data.ts` con builders puros (sin I/O) que devuelvan los objetos a insertar— usuario demo, preferences, categorías+subcategorías, 2 tarjetas, ≥10 transacciones (cada una con `id` explícito y `Decimal` para montos) e ≥3 ingresos. Sin tocar la DB; todo tipado contra los tipos de `@prisma/client`. Typechea solo.
3. Crear `prisma/seed.ts` que— (a) llame a `assertNotProdDatabaseUrl(process.env.DATABASE_URL)`, (b) haga los `upsert`/recreación idempotente dentro de una transacción usando los builders del paso 2, (c) loguee un resumen de cuántos registros creó y cierre la conexión. Typechea solo.
4. Agregar a `package.json` el bloque `"prisma": { "seed": "tsx prisma/seed.ts" }` y un script `"db:seed": "prisma db seed"`, sumando `tsx` a devDependencies; no modificar los scripts existentes. Typecheck y lint.
5. Agregar `prisma/__tests__/seed.test.ts` (vitest) que cubra el guard (rechaza `ep-floral-mud`, acepta la URL de dev) y la validez estructural de los builders (Transaction con `id`, `closingDay`/`dueDay` en rango, montos Decimal, unicidad de `name` de categoría por usuario). Tests + typecheck.

## Fuera de alcance

- Ejecutar el seed contra cualquier DB real como parte de la feature— la corrida real es un paso manual gated (ver Restricciones). El loop solo debe dejar el script escrito, tipado, lintado y con tests que NO tocan la base.
- Migraciones de schema, `prisma db push`/`migrate` o cualquier cambio en `prisma/schema.prisma`— el schema ya está materializado; el seed solo inserta datos.
- Poblar modelos secundarios/derivados (`BillingCycle`, `Installment`, `Payment`, `BankStatement`, `Alert`, `ThirdPartyPurchase`, `BudgetLimit`)— alcanza con User/Preferences/Category/SubCategory/CreditCard/Transaction/Income para testear el loop. Se pueden dejar como TODO.
- Cualquier UI, pantalla o cambio en `app/`— es un script de datos, sin superficie visual.
- Tocar el `.env` de Spensiv (que apunta a prod), `targets.json` crudo o el secreto `SPENSIV_DEV_DATABASE_URL`.

## Restricciones clave

- **Toca DB → gate humano requerido antes de correr el seed.** El script queda escrito y testeado por el loop, pero ejecutarlo contra la Neon dev (`db:seed`) es un paso manual que hace Augusto; el loop NO lo corre.
- **Nunca contra prod.** Prohibido escribir contra la Neon de producción `ep-floral-mud`. El seed DEBE abortar por sí mismo si la URL efectiva matchea ese patrón o si no hay URL de dev— defensa en profundidad además del guard del orquestador.
- **Idempotencia obligatoria**— correr el seed dos veces no puede fallar por unique constraints ni duplicar datos; usar `upsert` con claves deterministas o borrar+recrear el usuario demo en una transacción.
- **Campos reales del schema**— columnas camelCase sin `@map` de propiedad; respetar los `@@unique` (`User.email`, `User.clerkId`, `Category [userId,name]`, `SubCategory [categoryId,name]`) y el hecho de que `Transaction.id` no tiene `@default` (hay que proveerlo).
- **Montos con `Decimal`** (`decimal.js`), no floats nativos, para los campos `@db.Decimal(12,2)`.
- Sin dependencias nuevas salvo `tsx` (devDependency) como runner del seed en TS; nada más.
