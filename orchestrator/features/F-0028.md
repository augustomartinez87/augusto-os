## Contexto

= S-036 del backlog. Tres incidentes en 48h (2026-07-11/12, features F-0019, F-0020 y F-0025) que costaron reparación manual cada vez:

1. **STATE.json corrupto con null bytes trailing** (2 veces): `state.ts:51-53` — `saveState` hace `writeFileSync` directo; un kill/Ctrl+C a mitad de escritura deja el archivo con basura después del JSON válido y `JSON.parse` de `loadState` explota en la corrida siguiente. `renameSync` YA está importado en `state.ts:1` y no se usa — el fix canónico es escribir a `STATE.json.tmp` y renombrar encima. `loadState` además debe ser tolerante: extraer el primer valor JSON válido (estilo raw_decode), avisar por log y persistir la versión limpia.

2. **Pausa ciega de +5h** (2 veces): `limits.ts:33-48` — `parseResetTime` intenta `retry-after` y `resets at HH:MM`; si el output del CLI no trae ninguno (caso real observado), cae al fallback `ahora+5h` y `handleUsageLimit`/`index.ts:181` duermen hasta esa hora estimada aunque la ventana real del plan se libere antes. Augusto tenía tokens disponibles y el loop esperaba horas de más.

**Diseño del probe:** helper `probeAvailability()` en `limits.ts` que hace la llamada más barata posible al CLI de claude (modelo barato de `config/models.ts`, `--max-turns 1`, prompt trivial tipo "ok") y devuelve boolean según exit/status. Usarlo en dos puntos: (a) dentro de `handleUsageLimit` cuando NO hay hora parseada — loop de poll cada `PROBE_INTERVAL_MS` (sugerido 15 min, constante exportada) en vez de `sleepUntil(estimado)`; si hay hora parseada real, mantener `sleepUntil` + un probe de confirmación al despertar; (b) en el arranque (`index.ts`) cuando `pausedUntil` es futuro. Hacer el probe inyectable (patrón `callClaude?` ya usado en `architect.ts`/`planner.ts`/`reviewer.ts`) para testear sin CLI real.

REUSAR: strings de modelo centralizados en config (S-019), patrón de inyección para tests, suite existente de `state.ts` (S-021) como base para los tests nuevos.

## Pasos sugeridos

1. `saveState` atómico (tmp + renameSync) + `loadState` tolerante con reescritura; tests con fixture corrupto (JSON válido + `\0\0…` trailing).
2. `probeAvailability()` inyectable en `limits.ts` + reemplazo del fallback ciego por poll; conservar `parseResetTime` para los casos con hora real.
3. Probe al arranque con `pausedUntil` futuro en `index.ts`.
4. Tests de reanudación temprana (probe fake que falla N veces y luego pasa) y actualización de los tests existentes que asuman el sleep fijo.

## Fuera de alcance

- Routing multi-modelo del Builder (S-014) — el probe usa el modelo barato solo como ping, no cambia quién ejecuta steps.
- Formato/esquema de STATE.json (los `STATE.F-XXXX.archived.json` existentes deben seguir siendo legibles).
- Telegram/notificaciones (el aviso de pausa existente queda igual).

## Restricciones clave

- El probe debe ser de costo mínimo (1 llamada trivial por intervalo, nunca en paralelo) y respetar el gating por OPERATOR_STATE existente para notificaciones.
- Compatibilidad total con `npm start <feature>` en curso: un STATE sano nunca debe reescribirse de forma que pierda steps/commits.
- Sin dependencias nuevas.
