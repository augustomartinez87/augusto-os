## Contexto

El dashboard de control del orchestrator (`dashboard/index.html`, HTML estático single-file — decisión real de S-007, NO Next.js) ya tiene una `sec-nav` con pestañas secundarias (Backlog, Ideas, Log crudo) que se muestran/ocultan con un toggle genérico. S-016 (fast-follow de S-007) pide agregar **guías visuales de qué hacer en cada caso** (gate falla, deploy, etc.) como una pestaña más, para que Augusto no tenga que recordar de memoria el runbook cuando el sistema lo interpela desde el celu.

El loop debe **reusar** lo que ya existe en `dashboard/index.html`:
- El patrón de pestaña- botón `<button class="sec-link" data-t="…">` (líneas ~160-162) + panel `<div class="panel" id="p-…">` (líneas ~164-172).
- El toggle ya es genérico (`document.querySelectorAll(".sec-link")…` líneas ~421-426, arma `$("p-" + t.dataset.t)`): un `data-t` nuevo con su `#p-…` funciona **sin tocar JS**.
- Las CSS vars del theme y las clases de badge de estado (`.b-now`, `.b-done`, `.b-block`, `.b-hold`) ya definidas en el `<style>`; los diagramas deben tomar de ahí sus colores, no hardcodear hex.

El contenido de los flujos refleja procesos reales del sistema- el gate humano (`orchestrator/src/gates.ts`), los modos PRODUCT/OFFICE/SLEEP (`operator-state.ts` / OPERATOR_STATE.yaml), el review (`reviewer.ts`) y el auto-deploy en verde (ADR-0019). No inventar flujos que el sistema no ejecuta.

## Pasos sugeridos (el planner puede refinar)

1. Agregar en la `sec-nav` de `dashboard/index.html` (~línea 162) un `<button class="sec-link" data-t="flujos">Flujos</button>` y, junto a los otros paneles (~línea 172), un `<div class="panel" id="p-flujos">…</div>` vacío. Verificar que abre/cierra con el toggle existente sin JS nuevo.
2. Agregar en el `<style>` un bloque de clases `.flow-*` (nodo, conector, rama de decisión, etiqueta) que tomen sus colores de las CSS vars ya existentes (`--teal`, `--mut`, `--mut2`, `--panel`, badges `b-*`).
3. Construir el diagrama "Gate falla → aprobación humana" como markup inline (nodos con flechas): estado bloqueado → notificación (según modo OPERATOR_STATE) → decisión aprobar/rechazar → reanuda o queda en `blocked`.
4. Construir el diagrama "Deploy / release en verde": verde (tsc·lint·tests) → review → auto-deploy → aviso Telegram; rama de fallo → `release failed` → qué hacer.
5. Ajustar responsive- a 375px los nodos apilan en vertical y no desbordan (probar con DevTools o Playwright ya presente en `orchestrator/node_modules`).
6. Chequear que las otras pestañas (Backlog/Ideas/Log crudo) siguen funcionando igual y que abrir "Flujos" cierra las demás (comportamiento actual del toggle).

## Fuera de alcance

- Editar/crear diagramas desde la UI (son contenido estático versionado en el HTML).
- Realtime, polling o traer los diagramas desde Supabase/base de datos.
- Librerías de diagramas (Mermaid, D3, etc.) o cualquier asset externo/CDN — rompería el modelo single-file offline-safe.
- Migrar el dashboard a Next.js o introducir build step.
- Nuevos flujos más allá de los 2 mínimos (gate falla, deploy); ampliar el catálogo queda para un fast-follow.

## Restricciones clave

- **No toca DB, prod ni legal.** Es contenido estático en un HTML ya existente; no requiere gate humano.
- El dashboard es **HTML estático single-file** (`dashboard/index.html`) por decisión de S-007/RUNBOOK: prohibido agregar dependencias, bundlers, frameworks, `<script src>` externos o `<link>` a CDNs. Todo inline y autocontenido.
- Colores y tipografía **desde las CSS vars del theme** ya definidas; nada de hex nuevos sueltos que rompan el look en dark.
- **Mobile-first**- el dashboard se usa desde el celu; los diagramas deben ser legibles y sin scroll horizontal a 375px.
- No romper el toggle genérico de `sec-link`/`panel` ni las pestañas existentes; el `data-t` nuevo debe encajar en `$("p-" + t.dataset.t)` sin casos especiales.
