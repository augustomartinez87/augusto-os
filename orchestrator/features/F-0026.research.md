## Riesgos y restricciones técnicas para: Argos mobile — Bottom nav con pill prominente estilo mockup

**Resumen:** El bottom nav actual (MobileNav) tiene un diseño funcional con pill sutil (rounded-full + teal-dim) pero está fuertemente acoplado al layout (padding .pb-mobile-nav fijo en 3.5rem), a la configuración de navegación (NAV_SECTIONS con paths hardcodeados en PINNED) y al sistema de permisos (AuthContext). No existen tests automatizados, y cualquier cambio de altura del nuevo 'pill prominente' requiere sincronizar al menos 4 puntos: el componente MobileNav, la clase .pb-mobile-nav, el paddingBottom inline del nav, y la prop portfolioType en AppLayout. Se recomienda desacoplar la altura del nav a una variable CSS compartida y agregar tests antes del refactor.

- Iconos de lucide-react (LayoutDashboard, Wallet, PieChart, Zap, MoreHorizontal)

- Navegación con react-router-dom Link y useLocation

### Riesgos / Restricciones
- Coupling con layout: .pb-mobile-nav en index.css fija el padding-bottom a 3.5rem (h-14). Si el pill prominente cambia la altura del nav, hay que actualizar este valor sincronizadamente para evitar que el contenido quede tapado o con espacio muerto.
- Coupling con BottomSheet: el componente MobileNav renderiza BottomSheet para 'Más'. El pill prominente debe coexistir con el trigger de apertura del sheet sin romper la jerarquía z-index (nav z-50, sheet z-[100]).
- Rutas hardcodeadas: el array PINNED dentro de MobileNav contiene paths literales ('/portfolio/overview', etc.). Cualquier cambio en la navegación o en NAV_SECTIONS requiere mantenimiento manual en dos lugares.
- Acoplamiento fuerte con AuthContext y permisos: MobileNav depende del filtering vía getGroupedNavSections, que recibe isAdmin y allowedModules. Un cambio en la lógica de autorización impacta directamente qué ítems se muestran en el pill.
- Sin tests unitarios ni de integración: no se encontraron archivos de test para MobileNav (src/__tests__ vacío). Cualquier refactor del bottom nav no tiene red de seguridad automatizada.
- Estilos inline vs Tailwind: la implementación actual mezcla estilos inline (style={{...}}) con clases Tailwind. Un pill prominente requerirá coordinación para no duplicar ni pisar reglas CSS.
- Dark mode obligatorio: ThemeContext fuerza dark mode siempre. El diseño del pill debe validarse únicamente sobre fondo oscuro (#0C0C11).
- Safe area coupling: el nav usa paddingBottom: env(safe-area-inset-bottom) inline + la clase .pb-mobile-nav también usa env(safe-area-inset-bottom). Cualquier cambio de altura necesita ajustar ambos lugares.
